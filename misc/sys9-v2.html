How we think about agents

01

An agent is a loop with one gate

An agent takes a task + context and works however it likes. Whatever it does inside, the result must clear one fixed gate — your tests and reviews — before it counts as output. Because the gate never changes, quality is judged the same way every time. And every run is observed and fed back, so the next one improves — that’s what makes it trainable.

02

“Execution” is five parts — and two owners

“Execution” isn’t one thing — it’s five. You and your teams own how the work is shaped: the workflow and the pattern. The platform owns the rails that stay the same every time: the environment it runs in, the observability that records it, and the improvement loop that learns from it.

03

Workflow vs. Pattern — the two people mix up

These are the two words people confuse. Workflow is the order of phases — when things happen (a straight line). Pattern is how agents collaborate — who does what (a branching shape). The same workflow can run under very different patterns.

04

sys9 — the platform underneath

Those platform rails are concrete on sys9: every agent runs in its own run9 sandbox; db9 / fs9 / mem9 hold its state, files and memory; owl9 watches every run; model9 (soon) is the gateway to the models they call. You bring the workflow and the pattern — sys9 brings everything else.

Here’s the model running for real. I use Claude Code as the master agent — but instead of its built-in sub-agents, each one is a run9 sandbox running codex. The master plans a sprint (goals, acceptance criteria, task breakdown), then in /goal mode it drives the work, dispatching tasks to the run9 fleet by dependency and shipping each as a PR that must meet its acceptance criteria — the gate from beat 1.
