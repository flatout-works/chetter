run9 — Technical Design

A multi-tenant, serverless remote execution platform for coding agents.

run9 lets a user or an AI agent get an isolated Linux box on demand, run commands in it, save its disk as a snap, and fork a fresh box from any snap — with no servers to manage. It runs in production on AWS (Tokyo). The mental model is small: a box runs execs; a snap is the box's saved disk; a box always boots from a snap.
org → tenant & isolation unit box → an isolated container you run in exec → one run inside a box snap → the box's saved disk (fork-able)

0The big picture

control plane decides & remembers · data plane runs the boxes · storage is shared

The control plane (stateful, in Postgres) is separated from the data plane (stateless hosts). The single remote entry point on each host is run9vma, which dials out to the control plane over a long-lived gRPC stream.

1The sandbox: what runs a box

a box is an OCI container, not a virtual machine

A box is a Podman container on the crun runtime. run9ch ("run9 container helper") is a thin local CLI that binds a root filesystem into a temporary container and tears it down afterward. Isolation inside a host is standard Linux namespaces + cgroups. Tenants are kept apart at a stronger boundary: one org never shares an EC2 host with another, so the VM boundary separates orgs.

Podman + crun — OCI container JuiceFS FUSE — box rootfs (CoW) slirp4netns + nftables — network + egress lock cgroups v2 — CPU / memory run9ch-holder — static Go init / exec broker

2Serverless: how elasticity works

no servers to manage · pay for what you run · scale to zero

The elasticity-service keeps a global warm pool of pre-started EC2 hosts. When a box needs to run, it is placed with a simple ladder, always onto an org-exclusive host. Cold start targets ~1 second, because the host is already warm and the box boots from a snap instead of pulling an image.

AWS EC2 (c6g) — host fleet warm pool + bootstrap + idle reclaim org-exclusive hosts — hard tenant isolation gRPC — vma ↔ elasticity-service PostgreSQL — durable fleet state

3Snapshot & fork

a snap is a filesystem; fork is a metadata-only copy-on-write clone

A snap is an immutable snapshot of a box's disk filesystem only. Fork is cheap because it copies just the JuiceFS metadata directory, not the data — so it is effectively O(1). Parent and child share the same underlying objects in S3; each gets a fixed "owned epoch" so their later writes never collide.

End-to-end happy path
  • create — a box is placed on an org host; run9rt mounts the chosen snap as the box rootfs.
  • exec — commands run inside the Podman container; output streams back (foreground) or appends to a tape (background).
  • save — the box's current disk becomes a new immutable snap.
  • fork — juicefs clone meta/ makes a child snap instantly; a new box boots from it. The tree of snaps is tracked by parent_snap_id.
Two names that mislead: run9ch is "container helper" (Podman/crun), not cloud-hypervisor — there is no microVM. And run9tape stores exec output, not snapshots. The fast, cheap part of fork is that JuiceFS clones metadata only; the bytes stay shared in S3 until written.
