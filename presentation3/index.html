A small MCP control plane for orchestrating remote AI coding agent runners — durable task queues, triggers (cron + webhook), fleet health, and a built-in path into vulnerability scanning.

What & Why

Chetter is an MCP control plane that recruits AI agents as sub-contractors.

01General purpose

Any MCP-capable client (Claude, ChatGPT, OpenCode, curl) can talk to it. Not tied to one product.

02For AI, by AI

The control surface is tools, not a UI. An LLM is the typical operator.

03Remote, not local

Tasks run inside runner containers somewhere else. The MCP server holds no execution responsibilities.

04Durable queue

Tasks survive runner outages via DB-backed leases. Triggers (cron + PR review webhook) survive restarts via MySQL. No separate broker required.

05Observable fleet

Every task carries provider, model, variant, session id, and runner image digest for end-to-end traceability.

Schema-driven

Every tool's input and output is a Go struct with jsonschema tags. mcp.AddTool reflects at startup, so the JSON schema and the Go types cannot drift.

Two trigger types

cron triggers fire on a schedule; pr_review triggers fire on GitHub pull_request / issue_comment webhook events matching a repo. Both share the same prompt, agent, model, and timeout machinery.

Task Lifecycle

From MCP tool call to runner event, in one linear flow.

1. SubmitInsertTask with random task_id; status = pending; lease columns null
2. ClaimRunner polls ClaimTask via ConnectRPC; SELECT ... FOR UPDATE picks one pending task, sets lease + status = running
3. RunRunner spawns agent via harness interface, sends status via ReportTaskEvents RPC every heartbeat
4. Renew / ReapHeartbeat RPC renews lease; reaper reclaims expired leases and fails orphaned tasks
5. CompleteTerminal state (done, error, cancelled) written once; replay-safe. Cancel commands delivered via heartbeat response.

ConnectRPC Task Queue

Four RPCs replace the message broker. The database is the queue.

Runner lifecycle

Runners connect directly to the Chetter MCP server via ConnectRPC over HTTP.

RegisterRunner — announce presence Heartbeat — renew lease, get commands ClaimTask — poll pending tasks ReportTaskEvents — stream status updates lease renewal on each heartbeat

Database-backed queue

Tasks stored in chetter_tasks; SELECT ... FOR UPDATE with lease-based claiming.

pending → claimed → running lease via lease_expires_at column reaper reclaims expired leases cancel commands flow in HeartbeatResponse no separate broker to operate

Adding a third runner is just another container pointing at the same MCP server — no broker, no queue groups, no durable consumers.

Triggers

Unified trigger system: cron schedules and PR review webhooks share the same DB-backed machinery.

Five-field expressions and @descriptors for cron triggers. In-process, single-instance.

Webhook dispatch

Startup loads enabled pr_review triggers; webhook handler queries matching triggers by repo per event. /chetter-review comments are acknowledged via label + comment reply. No hardcoded reviewer config.

DB-backed config

trigger_type + trigger_config (JSON) columns on chetter_schedules eliminate the old hardcoded reviewer pattern. Add triggers at runtime.

Reaper & Health

Runner heartbeats every 5s; fleet health inferred from ConnectRPC activity.

ReapStaleTasksrunning tasks older than timeout + 120s grace → error
ReapExpiredLeasesleases past lease_expires_at → reclaimed; tasks → failed
GetRunnerFleetHealthjoins runner heartbeats; is_stale = last_seen_at > 60s ago
FleetActiverunning_tasks > stale_tasks (idle fleet looks dead)

Runners send heartbeats via ConnectRPC every 5s (configurable). Lease renewal is baked into the heartbeat response. No separate broker health to monitor.

Single process

Currently single-instance: cron entries, reaper, trigger dispatcher, and ConnectRPC endpoint live in-process. Two instances would double-trigger.

Stateless MCP

The HTTP transport uses Stateless mode, so the MCP server could be scaled — the rest of the system cannot, yet.

CI redeploy

.github/workflows/chetter.yml builds images, then curls a single-purpose Arcane webhook to redeploy.

Patterns & Tradeoffs

Choices that keep the codebase small and predictable.

Cancel via heartbeat

Cancel sets DB row to cancelled; runner receives cancel command in next HeartbeatResponse. No separate signal path needed.

Event write then ack

ReportTaskEvents writes to DB inside a transaction; error responses trigger runner retries. Idempotent updates keep replays safe.

Defensive env redaction

Keys matching SECRET|TOKEN|KEY|PASSWORD (case-insensitive) are scrubbed before both DB and wire paths. Users cannot smuggle secrets.

Timeouts on every call

ConnectRPC claim 30s poll, event handler 10s, schedule run 30s, Arcane 30s, HTTP read header 10s, shutdown 15s. No unbounded blocking.

Schema probes, not migrations

information_schema lookups at startup; ALTER TABLE IF NOT EXISTS for add-only schema evolution on a small footprint.

Trigger-based webhook dispatch

GitHub PR reviews no longer use hardcoded config. pr_review triggers in the DB are matched by repo at event time. Add, remove, or reconfigure reviewers without redeploying.

One file per concern

Technical Choices

Each technology is biased toward operational simplicity and at-least-once delivery.

MCP, not REST

The control surface is for AI agents, not humans. Tools are the native interface for any LLM client.

Streamable HTTP

Modern MCP transport. Stateless mode makes the server trivially load-balancer-friendly.

ConnectRPC

Task queue via server-streaming RPC over HTTP; lease-based claiming; cancel commands piggyback on heartbeats. No separate broker.

MySQL / TiDB

Same wire protocol as the rest of the stack. Small, add-only schema, conservative pool.

robfig/cron v3

Battle-tested, tiny, descriptor parser for @hourly-style. In-process is fine for single-instance.

Distroless binary

Multi-stage build → static-debian12:nonroot. No shell, no package manager, small surface.
