Self-hosted MCP control plane

Run AI coding agents like a small, observable fleet.

Chetter exposes an MCP tool surface so your AI agent — whether you use OpenCode, Claude Code, Pi, or another MCP client — can submit development work to remote runner containers. Docker Compose or Kubernetes. No message broker.

How it works

One Go service coordinates the work. Runners do the execution.

Chetter stays deliberately small: the MCP server accepts tool calls, persists task state to TiDB, serves it to runners via ConnectRPC, manages cron, PR review, issue, and event-callback automation, runs a data-driven model catalog with Git-backed definitions, hosts GitHub MCP tools with audit and artifact tracking, and ingests runner events. Runners poll for pending work, clone repositories, start OpenCode, Claude Code, or Pi through harnesses, stream progress, record token usage, and report results. An embedded SvelteKit web UI at port 18090 provides ConnectRPC-driven dashboards for tasks, triggers, sessions, artifacts, audit logs, and admin actions. Kubernetes-friendly with included manifests.

1. SubmitMCP tool call creates a pending task row in the database.
2. ClaimRunners poll the ConnectRPC endpoint; one claims the task via lease.
3. RunA runner starts OpenCode, Claude Code, or Pi through its harness, with Docker/gVisor isolation where supported.
4. ObserveEvents are persisted for status, progress, and audit trails.

Architecture

A boring stack on purpose.

client

OpenCode, Claude Code & Pi

The runner ships with integrated harnesses for OpenCode, Claude Code, and Pi. It runs stock CLIs behind harness adapters — OpenCode serve mode, Claude Code through a serve-proxy, and Pi through JSONL RPC — without inventing a custom agent runtime.

control plane

Go MCP server + web UI

A small Go process exposes /mcp, /healthz, and /webhook/github, registers 48 core schema-driven tools plus optional Arcane tools, manages triggers and event callbacks, orchestrates a data-driven model catalog from YAML or a Git definitions repo, hosts GitHub MCP tools with audit and artifact tracking, and records events. An embedded SvelteKit UI on a separate port provides ConnectRPC-backed dashboards for tasks, triggers, runners, sessions, artifacts, and audit logs.

execution

Plain Docker + optional gVisor

Agents run with normal Docker images and harness-specific execution paths. OpenCode and Claude Code get per-task Docker/gVisor containers; Pi runs through its RPC harness. The runner includes a gh wrapper that redirects write commands to Chetter MCP tools, plus image variants for Go, Python, Node, Rust, and minimal environments.

state

TiDB

Tables hold tasks with leases, raw events, triggers, trigger runs, event callbacks, agent sessions, checkpoints, runner heartbeats, teams, users, API tokens, definitions, proposal records, audit log events, task artifacts, full-text search columns, and token/cost totals. Secrets are redacted.

Why developers use it

Useful automation without a giant platform.

Remote, isolated execution

Keep agent work off your laptop. Runners execute inside Docker containers — optionally under gVisor sandboxing — with explicit CPU, memory, timeout, image, model, and provider settings.

Durable by default

Tasks survive runner restarts. Events are written before ack. Terminal states are updated idempotently. Timeouts and transport errors are classified as recoverable — failed sessions are automatically paused with workspace preservation, captured container diagnostics, and previous session export context, so you can resume from checkpoints on the same runner.

Triggers & schedules

Cron schedules, PR reviews, issue/comment responders, and task-event callbacks share DB-backed automation. PR review triggers have a built-in review template, issue triggers can filter labels and comments, and callbacks can create follow-up tasks or post webhook/Slack notifications.

Readable codebase

The main surface is a few focused Go packages: service orchestration, MCP tools, store, config, webhook handler, model catalog, definitions manager, event callbacks, error classification and diagnostics, full-text search helpers, and optional Arcane vulnerability scanning.

Standard protocols

Streamable HTTP MCP, ConnectRPC (HTTP/gRPC), TiDB (MySQL wire protocol), Docker Compose, and cron. No proprietary agent protocol required.

Built for operators

Inspect status, logs, latest events, container diagnostics for transport failures, progress summaries, fleet health, runner image digests, model IDs, session exports, token totals, audit log events, trigger runs, and task-created artifacts from MCP and the web UI.

Tool surface

Everything is exposed as MCP tools.

Chetter is designed for AI clients first. Tool schemas are generated from Go structs, so implementation and JSON contracts stay aligned. The always-on surface is 48 tools; Arcane adds five more when configured.

Quick start

Bring up the control plane and two runners with Compose.

Clone the repo, configure tokens, start the stack, then point your MCP client at /mcp. The web UI and ConnectRPC API are available on port 18090. Use a GITHUB_TOKEN when runners need private repo access or pull request permissions.

shell
git clone https://github.com/flatout-works/chetter.git
cd chetter
cp .env.example .env
./deploy/build.sh

docker compose --env-file .env \
  -f deploy/compose.yaml \
  -f deploy/compose.local.yaml up -d

curl http://localhost:18088/healthz

Deployment shape

Small enough to understand, useful enough to automate real work.

chetter-mcpGo server: MCP tools (port 18088) + web UI (port 18090) + ConnectRPC + triggers + webhook + model catalog
chetter-runnerplain Dockerfile-based environments with OpenCode, Claude Code, and Pi harnesses; Docker/gVisor isolation via USE_GVISOR=true where supported
TiDBtask state, events, triggers, sessions, definitions, runners, auth, audit log, artifacts

Comes with Docker Compose and Kubernetes manifests. Production can use an external TiDB instance by setting DATABASE_DSN. Enable kernel-level sandboxing by installing gVisor (runsc) on the Docker host and setting USE_GVISOR=true. Sync model catalogs, agents, skills, triggers, and task templates from a Git definitions repo via DEFINITIONS_REPO.
