Self-hosted MCP control plane

Run AI coding agents like a small, observable fleet.

Chetter exposes an MCP tool surface so your AI agent — whether you use OpenCode, Claude Code, or another MCP client — can submit development work to remote containerized runners with optional gVisor sandboxing. An embedded SvelteKit web UI (Flowbite components) and ConnectRPC API give humans and tools a dashboard for tasks, triggers, runners, sessions, definitions, and artifacts. One server. Docker Compose or Kubernetes. No message broker.

How it works

One Go service coordinates the work. Runners do the execution.

Chetter stays deliberately small: the MCP server accepts tool calls, persists task state to TiDB, serves it to runners via ConnectRPC, manages triggers (cron schedules and webhook events), runs a data-driven model catalog with Git-backed definitions sync and change proposals, hosts GitHub MCP tools with audit and artifact tracking, processes event callbacks, and ingests runner events. Runners poll for pending work, clone repositories, start OpenCode or Claude Code inside plain Docker containers (optionally under gVisor sandboxing), stream progress, and report results. An embedded SvelteKit web UI with Flowbite components at port 18090 provides ConnectRPC-driven dashboards for tasks, triggers, sessions, artifacts, definitions, and admin actions. Kubernetes-friendly with included manifests.

1. SubmitMCP tool call creates a pending task row in the database.
2. ClaimRunners poll the ConnectRPC endpoint; one claims the task via lease.
3. RunA runner starts OpenCode or Claude Code inside a Docker container (with optional gVisor sandboxing).
4. ObserveEvents are persisted for status, progress, and audit trails.

Architecture

A boring stack on purpose.

client

OpenCode & Claude Code

The runner ships with integrated harnesses for the two leading open-source AI agent tools. Both run the stock CLI — no forks, no custom agent protocol. Just standard serve mode or batch execution inside Docker containers.

control plane

Go MCP server + web UI

A small Go process exposes /mcp, /healthz, and /webhook/github, registers schema-driven tools (52 total), manages triggers (cron + webhook), orchestrates a data-driven model catalog synced from a Git definitions repo, hosts GitHub MCP tools with audit and artifact tracking, processes event callbacks, and records events. An embedded SvelteKit UI with Flowbite components on a separate port provides ConnectRPC-backed dashboards for tasks, triggers, runners, sessions, definitions, and artifacts.

execution

Plain Docker + optional gVisor

Agents run inside standard Docker containers built from normal Dockerfiles. The runner ships with integrated harnesses for OpenCode and Claude Code, a gh wrapper that redirects write commands and gh api calls to Chetter MCP tools, and image variants for Go, Python, Node, Rust, and minimal environments. Enable kernel-level sandboxing with USE_GVISOR=true and the runsc runtime. Application-level session resume lets agents pause and continue work on the same runner.

state

TiDB

Tables hold tasks (with leases), task events, triggers (with trigger_type + trigger_config), trigger runs, runners, teams, users, API tokens, agent sessions, model catalogs, audit log events, task artifacts, event callbacks, definition sources, definitions, definition sync runs, and definition change proposals. Secrets are redacted.

Why developers use it

Useful automation without a giant platform.

Remote, isolated execution

Keep agent work off your laptop. Runners execute inside Docker containers — optionally under gVisor sandboxing — with explicit CPU, memory, timeout, image, model, and provider settings.

Durable by default

Tasks survive runner restarts. Events are written before ack. Terminal task states are updated idempotently to make replays safe.

Triggers & automation

Cron triggers and webhook-triggered PR reviews share the same DB-backed trigger system. PR review triggers have a built-in review template, so prompt is optional. The web UI shows trigger run history with links to task detail pages. Nightly docs, changelogs, audits, scans, and automated code reviews all fit naturally. Triggers and model catalogs can be synced from a Git definitions repo with periodic auto-sync.

Readable codebase

The main surface is a few focused Go packages: service orchestration, MCP tools, store, config, webhook handler, model catalog, definitions manager, event callbacks, and optional Arcane vulnerability scanning.

Standard protocols

Streamable HTTP MCP, ConnectRPC (HTTP/gRPC), TiDB (MySQL wire protocol), Docker Compose, and cron. No proprietary agent protocol required.

Built for operators

Inspect status, logs, latest events, progress summaries, fleet health, runner image digests, model IDs, session exports, audit log events, task-created artifacts, and TiDB quota exhaustion warnings from MCP and the web UI.

Tool surface

Everything is exposed as MCP tools.

Chetter is designed for AI clients first. Tool schemas are generated from Go structs, so the implementation and JSON contracts stay aligned.

Quick start

Bring up the control plane and two runners with Compose.

Clone the repo, configure tokens, start the stack, then point your MCP client at /mcp. The MCP server is on port 18088; the web UI and ConnectRPC API are on port 18090. Use a GITHUB_TOKEN when runners need private repo access or pull request permissions.

shell
git clone https://github.com/flatout-works/chetter.git
cd chetter
cp .env.example .env

docker compose --env-file .env \
  -f deploy/compose.yaml \
  -f deploy/compose.local.yaml up -d

curl http://localhost:18088/healthz

Deployment shape

Small enough to understand, useful enough to automate real work.

chetter-mcpGo server: MCP tools (port 18088) + web UI (port 18090) + ConnectRPC + triggers + webhook + model catalog + definitions + event callbacks
chetter-runnerplain Dockerfile-based containers with OpenCode & Claude Code harnesses, gh wrapper blocking writes + gh api, optionally gVisor-sandboxed via USE_GVISOR=true
TiDBtask state, events, triggers, trigger runs, runners, sessions, auth, audit log, artifacts, definitions, event callbacks

Comes with Docker Compose and Kubernetes manifests. Production can use an external TiDB instance by setting DATABASE_DSN. Enable kernel-level sandboxing by installing gVisor (runsc) on the Docker host and setting USE_GVISOR=true. Sync triggers, model catalogs, and agent configs from a Git definitions repo via DEFINITIONS_REPO. The web UI detects TiDB quota exhaustion and shows an operator banner.
