Technical tour

One control plane, many disposable runners.

Chetter is intentionally boring: one Go server, one SQL database, a ConnectRPC runner protocol, and ordinary container images. The interesting part is how these pieces combine into durable agent automation.

System map

The server owns state. Runners own execution.

This split keeps the control plane understandable and lets you scale execution by adding runner containers or Kubernetes workloads.

Server

Go MCP + web UI

Exposes /mcp, GitHub webhooks, ConnectRPC endpoints, trigger scheduling, event callbacks, managed Git identities, model catalog, definitions sync (scoped to global/, groups/, repos/), MCP endpoint resolution, agent catalog API, execution attempt tracking, audit log, and embedded SvelteKit UI.

Database

TiDB, MySQL & PostgreSQL

Stores tasks, leases, events, triggers, sessions, checkpoints, runners, teams, API tokens, git identities, definitions, audit events, artifacts, and searchable text.

Runner

Harness controller

Claims work, prepares a workspace, starts the chosen harness, supervises timeouts, streams events, and cleans up containers and workspaces.

Agent image

Project runtime

Language/tooling images inherit from the agent base and run OpenCode, Claude Code, Codex, Pi, or CodeWhale against the mounted workspace.

MCP

AI-facing control surface

Tasks, triggers, sessions, runner health, GitHub artifacts, definitions, aggregate cost and token summaries, and admin actions are exposed as schema-driven MCP tools.

ConnectRPC

Runner protocol

Runners use RegisterRunner, Heartbeat, ClaimTask, and ReportTaskEvents. Cancel and drain commands return on heartbeat responses.

Web UI

Human operations

The UI uses ConnectRPC to show tasks, triggers, runners, sessions, audit events, artifacts, agent definitions (browsable catalog with per-agent activity), and aggregate task activity with server-side team/repo filtering. Task pages surface full configuration — provider, model, harness, session mode, git ref, skills, and environment — plus deadline extension with URL state persistence.

Queue semantics

The task queue is just SQL plus leases.

No separate broker is required. The database is the source of truth for pending work, ownership, attempts, terminal state, and event history.

A
Claim safely

SELECT ... FOR UPDATE SKIP LOCKED finds claimable work. The winner sets status=running, runner_id, claimed_at, started_at, and a lease expiry.

B
Renew cheaply

Heartbeats refresh leases for the runner's current task ids and return commands such as cancel or drain.

C
Recover predictably

The reaper runs every 30 seconds. Expired leases create a fresh execution attempt on reclaim, restarting the agent session with the previous attempt's context. Tasks that exhaust max_attempts (default 5) are marked error with a timeout category. Each execution attempt is a separate row in the execution_attempts table, giving operators a clear retry trail. The reaper recovers from panics so one bad pass never kills the loop, and it stamps a lastReapAt health signal on /api/server-info so operators can spot a stalled reaper.

D
Extend deadlines

Operators can add time to a pending or running task from the web UI. ExtendTask bumps timeout_sec in place and records a task_timeout_extended audit event — no restart or resubmission required.

E
Pause and resume

Tasks with session_mode: resumable create a checkpoint-backed session. Sessions are task-owned and track each user prompt in sequence. chetter_resume_agent_session appends a follow-up prompt and queues a new execution attempt in the same session. chetter_recover_task recovers terminal tasks in fresh agent sessions using the previous session export as context — each recovery creates a new execution attempt, preserving the full retry history.

Execution modes

Serve mode and RPC mode

OpenCode, Claude Code (serve-proxy), CodeWhale, and Codex (App Server proxy) harnesses expose an HTTP serve API for streaming, session management, and prompt dispatch. OpenCode implements CompletionAwareHarness — it dispatches prompts asynchronously, detects session completion via SSE session.status idle events, and falls back to polling, so long-running tasks survive proxy timeouts or network drops. Pi uses RPC mode with JSONL events over stdio. Streaming events are accumulated and flushed in short publish windows, and the runner normalizes every harness into the same task events. A progress watchdog detects and recovers stalled harnesses that stop producing events, keeping tasks resilient even when the agent process hangs.

Isolation

Workspace + container boundary

Docker mode bind-mounts the workspace into /workspace. gVisor can run containers with runsc. Transparent HTTP and DNS proxies route all outbound traffic through the runner, and gVisor DNS is forwarded so endpoint hostnames resolve inside sandboxed containers.

Diagnostics

Failures keep evidence

Transport failures capture container state, HTTP probes, and log tails. Recoverable failures can preserve workspaces and session exports for follow-up tasks.

Definitions

Scoped Git-backed configuration

Agent markdown, skills, triggers, task templates, MCP endpoints, and model catalogs sync from Git every five minutes. Definitions are organized under explicit scope directories — global/, groups/<team>/, repos/<owner>/<repo>/ — that control visibility. Global definitions are available to all teams; team-scoped definitions are only visible to that team's tokens. Definitions are materialized and validated before use.

Model catalog

Provider details stay data-driven

The catalog resolves provider id, model id, base URL, API key env var, provider kind (openai_compatible, native, aws_bedrock), and per-harness API transport before the task reaches a runner. One logical provider — a LiteLLM router, for example — can map to different wire protocols per harness (OpenAI completions, Anthropic messages, or Responses API) with no code changes.

Cost & Audit

Full audit trail with cost tracking

Every server action is logged with event type, source, target, repo, token identity, and searchable detail. Aggregate token usage and cost summaries group by team, trigger, or repo across configurable time windows. The web UI filters, toggles, and searches the trail in real time.

MCP Endpoints

External tool integration

Agent definitions reference named MCP endpoint definitions synced from the config repo. At claim time the server resolves URLs, headers, and bearer-token credentials, then delivers endpoint tokens to the runner. Each harness — OpenCode, Claude Code, CodeWhale, Pi, and Codex — receives native MCP configuration. Tokens are validated and protected from task overrides, keeping secrets out of task payloads. Chetter MCP traffic routes through a runner-local reverse proxy so task containers access the server at a consistent endpoint; the relay authenticates internal requests, and OpenCode re-applies the runner-configured server settings after cloning the repository to prevent config hijacking. OAuth is explicitly disabled for the Chetter MCP connection so non-interactive runners never block on an OAuth flow. gVisor DNS is routed through the runner so endpoint hostnames resolve inside sandboxed containers.
