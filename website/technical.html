Technical Architecture

How Chetter is built.

A small MCP control plane for orchestrating remote AI coding agent runners — durable task queues, cron/PR/issue triggers, task-event callbacks, GitHub MCP tools, model catalogs, definitions sync, searchable observability, recovery flows, fleet health, web UI, and optional vulnerability scanning.

Go + SvelteKit Streamable HTTP MCP ConnectRPC TiDB GitHub MCP tools Issue triggers Event callbacks Model Catalog Full-text search Arcane Trivy

What & Why

Chetter is an MCP control plane that recruits AI agents as sub-contractors.

01General purpose

Any MCP-capable client (Claude, OpenCode, ChatGPT, curl) or browser (web UI) can talk to it. Runners can execute OpenCode, Claude Code, or Pi harnesses.

02For AI, by AI

The control surface is tools, backed by a web UI for humans. An LLM is the typical operator.

03Remote, not local

Tasks run inside runner containers somewhere else. The MCP server holds no execution responsibilities.

04Durable queue

Tasks survive runner outages via DB-backed leases. Triggers, event callbacks, sessions, checkpoints, and recovery context survive restarts via TiDB. No separate broker required.

05Observable fleet

Every task carries provider, model, variant, session id, runner image digest, token totals, and cost totals for traceability. Audit log and artifact tracking record server-side events.

Schema-driven

Every tool's input and output is a Go struct with jsonschema tags. mcp.AddTool reflects at startup, so the JSON schema and the Go types cannot drift.

Three trigger types

cron triggers fire on a schedule; pr_review triggers fire on GitHub pull_request / issue_comment events; issue triggers fire on opened/reopened issues or issue comments, with optional label filtering. All trigger types share prompt, agent, harness, model, timeout, session, and run-history machinery.

Task Lifecycle

From MCP tool call to runner event, in one linear flow.

1. SubmitInsertTask with random task_id; status = pending; lease columns null. Or resume an existing agent session.
2. ClaimRunner polls ClaimTask via ConnectRPC; SELECT ... FOR UPDATE picks one pending task, sets lease + status = running
3. RunRunner spawns the selected harness (OpenCode, Claude Code, or Pi), sends status via ReportTaskEvents RPC, tracks token usage, and redirects GitHub writes to Chetter MCP tools.
4. Renew / Reap / DrainHeartbeat RPC renews lease; reaper reclaims expired leases and fails orphaned tasks; drain requests stop new claims and exit after in-flight work completes.
5. CompleteTerminal state (done, error, cancelled) written once; replay-safe. Cancel and drain commands delivered via heartbeat response.

ConnectRPC Task Queue

Four RPCs replace the message broker. The database is the queue.

Runner lifecycle

Runners connect directly to the Chetter MCP server via ConnectRPC over HTTP. A dedicated CHETTER_RUNNER_RPC_TOKEN authenticates runner RPC calls.

RegisterRunner — announce presence Heartbeat — renew lease, get commands (cancel, drain) ClaimTask — poll pending tasks ReportTaskEvents — stream status updates lease renewal on each heartbeat

Database-backed queue

Tasks stored in chetter_tasks; SELECT ... FOR UPDATE with lease-based claiming.

pending → claimed → running lease via lease_expires_at column reaper reclaims expired leases (30s interval, 120s grace) cancel + drain commands flow in HeartbeatResponse no separate broker to operate

Adding a third runner is just another container pointing at the same MCP server — no broker, no queue groups, no durable consumers.

Triggers

Unified trigger system: cron, PR review, and issue automation share the same DB-backed machinery.

Five-field expressions and @descriptors for cron triggers. In-process, single-instance.

Webhook dispatch

GitHub webhooks query enabled PR review and issue triggers by repo at event time. Bot comments are skipped unless a trigger opts in with bot_comments:true.

DB-backed config

trigger_type plus JSON trigger_config drives cron, PR review, and issue behavior without hardcoded reviewer configuration.

Trigger run history

The web UI shows trigger runs with links to task detail pages, token totals, and task-created artifacts for traceability.

Definitions in Git

Optionally sync model catalogs, agents, skills, triggers, and task templates from a Git definitions repo via DEFINITIONS_REPO with periodic auto-sync every 5 minutes.

Reaper & Health

Runner heartbeats every 5s; fleet health inferred from ConnectRPC activity.

ReapStaleTasksrunning tasks older than timeout + 120s grace → error
ReapExpiredLeasesleases past lease_expires_at → reclaimed; tasks → failed
DrainRunnerMCP tool requests a runner to stop claiming, finish in-flight work, then exit. Runner reports draining status on heartbeats.
GetRunnerFleetHealthjoins runner heartbeats; is_stale = last_seen_at > 60s ago
FleetActiverunning_tasks > stale_tasks (idle fleet looks dead)

Runners send heartbeats via ConnectRPC every 5s (configurable). Lease renewal is baked into the heartbeat response. No separate broker health to monitor.

Stateless MCP

The HTTP transport uses Stateless mode, so the MCP server could be scaled — the rest of the system cannot, yet.

Patterns & Tradeoffs

Choices that keep the codebase small and predictable.

Cancel via heartbeat

Cancel sets DB row to cancelled; runner receives cancel command in next HeartbeatResponse. No separate signal path needed.

Drain via heartbeat

DrainRunner sets a flag; runner receives drain command in next HeartbeatResponse, stops claiming new work, finishes in-flight tasks, then exits.

Event write then ack

ReportTaskEvents writes to DB inside a transaction; error responses trigger runner retries. Idempotent updates keep replays safe.

Defensive env redaction

Keys matching SECRET|TOKEN|KEY|PASSWORD (case-insensitive) are scrubbed before both DB and wire paths.

Timeouts on every call

ConnectRPC claim 30s poll, event handler 10s, trigger run 30s, Arcane 30s, HTTP read header 10s, shutdown 15s.

Schema probes, not migrations

information_schema lookups at startup; ALTER TABLE IF NOT EXISTS for add-only schema evolution on a small footprint.

Config in Git

Model catalog, agents, skills, triggers, and task templates can be synced from a Git definitions repo via DEFINITIONS_REPO, with periodic auto-sync every 5 minutes and proposal PR tools for changes.

Trigger-based webhook dispatch

GitHub PR reviews and issue automation no longer use hardcoded config. Triggers in the DB are matched by repo and event at webhook time.

Server-side signatures

GitHub MCP tools append a canonical single-line Chetter signature footer to every created artifact, enabling passive artifact discovery and tracking.

One file per concern

service.go orchestration, tools.go MCP, store.go SQL, runner_rpc.go ConnectRPC, catalog.go model catalog, definitions.go, event_callbacks.go, arcane.go HTTP.

Technical Choices

Each technology is biased toward operational simplicity and at-least-once delivery.

MCP, not REST

The control surface is for AI agents, not humans. Tools are the native interface for any LLM client.

Streamable HTTP

Modern MCP transport. Stateless mode makes the server trivially load-balancer-friendly.

ConnectRPC

Task queue via long-polling RPC over HTTP; lease-based claiming; cancel and drain commands piggyback on heartbeats. No separate broker.

TiDB

MySQL wire protocol. Small, add-only schema, FULLTEXT indexes for table search, conservative pool, and room for vector search and HTAP on the roadmap.

robfig/cron v3

Battle-tested, tiny, descriptor parser for @hourly-style. In-process is fine for single-instance.

Data-driven model catalog

YAML-based pkg/modelcatalog replaces hardcoded provider config. Supports DeepSeek, Z.ai, Anthropic, Synthetic (GLM-5.2), and OpenCode Zen (deepseek-v4-flash-free) with per-harness overrides.

Server image

Multi-stage Go build on debian:bookworm-slim, with git and openssh-client available for definitions repository clone/pull operations.
