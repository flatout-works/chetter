Technical tour

One control plane, many disposable runners.

Chetter is intentionally boring: one Go server, one SQL database, a ConnectRPC runner protocol, and ordinary container images. The interesting part is how these pieces combine into durable agent automation.

System map

The server owns state. Runners own execution.

This split keeps the control plane understandable and lets you scale execution by adding runner containers or Kubernetes workloads.

Server

Go MCP + web UI

Exposes /mcp, GitHub webhooks, ConnectRPC endpoints, trigger scheduling, event callbacks, model catalog, definitions sync, audit log, and embedded SvelteKit UI.

Database

TiDB / MySQL

Stores tasks, leases, events, triggers, sessions, checkpoints, runners, teams, API tokens, definitions, audit events, artifacts, and searchable text.

Runner

Harness controller

Claims work, prepares a workspace, starts the chosen harness, supervises timeouts, streams events, and cleans up containers and workspaces.

Agent image

Project runtime

Language/tooling images inherit from the agent base and run OpenCode, Claude Code, Pi, or CodeWhale against the mounted workspace.

MCP

AI-facing control surface

Tasks, triggers, sessions, runner health, GitHub artifacts, definitions, and admin actions are exposed as schema-driven MCP tools.

ConnectRPC

Runner protocol

Runners use RegisterRunner, Heartbeat, ClaimTask, and ReportTaskEvents. Cancel and drain commands return on heartbeat responses.

Web UI

Human operations

The UI uses ConnectRPC to show tasks, triggers, runners, sessions, audit events, artifacts, team/repo filters, and recovery actions.

Queue semantics

The task queue is just SQL plus leases.

No separate broker is required. The database is the source of truth for pending work, ownership, attempts, terminal state, and event history.

A
Claim safely

SELECT ... FOR UPDATE SKIP LOCKED finds claimable work. The winner sets status=running, runner_id, claimed_at, started_at, and a lease expiry.

B
Renew cheaply

Heartbeats refresh leases for the runner's current task ids and return commands such as cancel or drain.

C
Recover predictably

The reaper runs every 30 seconds. Expired leases are reclaimed to pending while attempts remain; otherwise they become error with a timeout category.

Execution modes

Serve mode and RPC mode

OpenCode, Claude Code (serve-proxy), and CodeWhale harnesses expose an HTTP serve API for streaming, session management, and prompt dispatch. Pi uses RPC mode with JSONL events over stdio. The runner normalizes both into task events.

Isolation

Workspace + container boundary

Docker mode bind-mounts the workspace into /workspace. gVisor can run containers with runsc, and proxy/DNS settings constrain outbound network access.

Diagnostics

Failures keep evidence

Transport failures capture container state, HTTP probes, and log tails. Recoverable failures can preserve workspaces and session exports for follow-up tasks.

Definitions

Git-backed configuration

Agent markdown, skills, triggers, task templates, and model catalogs can sync from Git every five minutes. Definitions are materialized and validated before use.

Model catalog

Provider details stay data-driven

The catalog resolves provider id, model id, base URL, API key env var, and per-harness overrides before the task reaches a runner.

Artifacts

GitHub output is auditable

Server-side Chetter signatures on issues, comments, PRs, and reviews make task-created artifacts discoverable and auditable later.
