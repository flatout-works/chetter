Chetter

Self-hosted MCP server for running autonomous AI development agents on a fleet of containerized runners.

  • Submit tasks against Git repos with LLM agents in isolated containers
  • Track progress — streaming events, logs, session transcripts
  • Automate — cron triggers, PR review webhooks, issue/comment responders
  • Resume — pause sessions and continue on the same runner
  • Powered by TiDB

Press → or click Next to begin.

Design Principles

1
Standard harnesses. Delegate execution to existing CLI tools — OpenCode, Claude Code, Pi. No custom agent runtime.
2
Deploy anywhere. Server and runners run on plain Docker or Kubernetes. Only need a TiDB database.
3
GitHub-native orchestration. PRs, issues, reviews, and comments drive agent workflows.
4
Plain containers as environments. You define the image with the stack your project needs.
5
MCP-first, API+UI for observation. All interaction flows through MCP tools. The API and web UI are for monitoring.

Architecture Overview

Control Plane

  • main.go — MCP server entry point
  • internal/service — business logic, tools, triggers
  • internal/store — TiDB schema & migrations
  • internal/webhook — GitHub webhook handler
  • internal/webapi — ConnectRPC + streaming

Database

  • TiDB — MySQL wire protocol
  • Tasks, events, sessions, artifacts, audit log
  • Goose migrations + sqlc generated queries

Runner Fleet

  • runner/ — separate Go module
  • Polls ClaimTask via ConnectRPC
  • Spawns Docker containers per task
  • Supports gVisor sandboxing (runsc)

Agent Harness

  • OpenCode, Claude Code, Pi
  • Runs as HTTP server inside container
  • Runner sends prompt, streams events back

The Database

Two sources of truth for schema evolution

Runtime Schema (internal/store/schema.go)

ApplySchema() runs on every server startup. Provides zero-downtime auto-migration for new columns via ensure*Columns helpers.

Versioned Migrations (db/migrations/*.sql)

Goose runs explicit migrations. sqlc reads all migrations to infer the schema for Go code generation.

Core Tables

  • chetter_tasks — task state, config, exports, error category
  • chetter_task_events — streaming progress + heartbeats
  • chetter_agent_sessions — session lifecycle, pause/resume
  • chetter_session_runs — individual run attempts
  • chetter_task_artifacts — GitHub issues/PRs/comments created
  • chetter_audit_log — server-side event log

Task Lifecycle — Overview

Create ──► Queue ──► Claim ──► Lease ──► Execute ──► Complete
  │          │          │          │            │
  ▼          ▼          ▼          ▼            ▼
MCP/      TiDB      Runner    Heartbeat      Events
Webhook             Poll       Renew         + Export

A task moves from creation to completion through a small number of well-defined states. The runner fleet is demand-driven: runners claim work, not the other way around.

Step 1 — Task Creation

Entry Points

  • MCP tool — chetter_submit_task
  • Cron trigger — periodic schedules
  • PR review webhook — new pull request opened
  • Issue/comment webhook — labeled or commented
  • Session resume — follow-up prompt on paused session

What Gets Stored

  • prompt, git_url, git_ref
  • agent_image, harness, timeout_sec
  • skills, env (issue number, GitHub token, etc.)
  • team_id — auto-stamped from auth context
  • trigger_name / trigger_type — when fired by a trigger

Tasks start in pending status. The reaper and claim logic both operate on this queue.

Step 2 — Queuing & Claiming

Atomic Claim

Runners poll ClaimTask over ConnectRPC. The server runs: 

SELECT ... FROM chetter_tasks
WHERE status = 'pending'
ORDER BY created_at
FOR UPDATE SKIP LOCKED
LIMIT 1

This guarantees exactly one runner claims a task, even under contention.

Lease Model

  • Claim sets claimed_at and lease_expires_at
  • Default lease: 60 seconds
  • Runner renews lease via periodic Heartbeat
  • If a runner dies, the lease expires and the task becomes claimable again

Failed tasks retry up to max_attempts (default 5). Each retry increments attempt.

Step 3 — Runner Lease & Heartbeat

Heartbeat Flow

1
Runner sends Heartbeat(taskID) every ~30s
2
Server extends lease_expires_at
3
Server updates last_event_at
4
If heartbeat stops, the Reaper (30s loop) reclaims the task

Reaper Rules

  • Lease expired + no heartbeat → mark error, "lease expired"
  • Task exceeded timeout_sec → mark error, "timeout"
  • Session paused but TTL expired → mark expired
  • reaperHealthMaxEventSec = 120 — tasks silent longer than this are stale

Step 4 — Agent Session & Prompt

Inside the Container

  1. Runner starts harness in serve mode:
    opencode serve --port 9999
  2. Runner waits for /config to respond (up to 120s)
  3. Runner POST /session to create a session
  4. Runner POST /session/{id}/message with the task prompt
  5. Harness (OpenCode) runs the agent against the cloned repo

Environment Injected

  • ISSUE_NUMBER, ISSUE_TITLE, ISSUE_BODY
  • GITHUB_TOKEN — App installation token
  • GITHUB_REPO, CHETTER_TASK_ID
  • WORKSPACE=/workspace
  • Any env vars from the trigger/task config

The prompt includes skill hints and the agent definition loaded from the configured definitions repo.

Step 5 — Event Streaming

Event Types

  • task.progress — agent thinking, tool calls, file edits
  • task.heartbeat — throttled to 1 stored event/min per task
  • task.failed.{category} — error with classification
  • task.completed — success

Events are stored in chetter_task_events as JSON payloads.

What You See

  • Web UI live stream
  • chetter_task_events MCP tool
  • chetter_task_progress — distilled timeline
  • Runner stdout/stderr piped to container logs

Heartbeat events update the lease but are rate-limited in storage to avoid table bloat.

Step 6 — Completion, Failure & Recovery

Terminal States

  • done — agent finished normally
  • error — failure with error_category
  • cancelled — user or system cancellation

Error Categories

  • timeout, transport_error, runtime_error
  • model_error, budget_exceeded, stuck

Recoverable Sessions

If a task fails with timeout or transport_error and the session mode is resumable:

  1. Runner preserves the workspace
  2. Server marks the session recoverable
  3. A follow-up prompt can resume on the same runner
  4. Session export is saved as a transcript for context

Triggers & Automation

Cron Triggers

  • Defined as YAML in the definitions repo
  • Synced via chetter_sync_definitions
  • Examples: nightly docs update, vulnerability scan, changelog

PR Review Trigger

  • Watches a repo for new pull requests
  • Fires an agent to review code
  • Can pause for follow-up feedback

Issue Triggers

  • issue type with event and match_labels
  • Example: implementer runs when approved label is added
  • Bot-comment filtering skips the app’s own comments by default

Webhooks

  • GitHub App webhook → handler → trigger match → task submit
  • Optional: configure GITHUB_APP_* env vars

Auth & Multi-Tenancy

Admin Token

CHETTER_MCP_AUTH_TOKEN env var. Single static bearer token. Bypasses all scoping.

Team Tokens

  • Stored in api_tokens table (SHA-256 hashed)
  • Each token belongs to a user in a team
  • Managed via chetterctl CLI

Scoping

  • Tasks/schedules get team_id auto-stamped
  • Team-scoped tokens see only their team's work
  • List queries filter by team_id
  • Audit log tracks who did what

Web UI & Observability

Web UI (web/)

  • SvelteKit + Flowbite-Svelte
  • Task list, detail, events, progress timeline
  • Session list, runs, checkpoints
  • Trigger management
  • Audit log viewer

MCP Tools for Inspection

  • chetter_task_status
  • chetter_task_events / chetter_task_progress
  • chetter_task_export — full session transcript
  • chetter_list_audit_events
  • chetter_list_task_artifacts — GitHub artifacts
  • chetter_runner_health — fleet status

Summary

1
Tasks are created via MCP, triggers, or webhooks and stored in TiDB.
2
Runners claim tasks atomically with a 60-second lease.
3
The runner spawns a container, starts the harness, and sends the prompt.
4
Events stream back in real time while the runner renews the lease.
5
On completion or failure, the runner publishes results and the server stores the transcript.
6
Resumable sessions can be paused and continued with follow-up prompts.
