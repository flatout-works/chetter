Developer architecture deck

Chetter turns agent work into a visible, resumable system.

It is not a custom agent brain. It is a control plane for standard coding harnesses running inside isolated dev containers.

Start with the map

The four big parts

Control

Server + TiDB hold the queue, leases, sessions, events, triggers, definitions, teams, and audit trail.

Configuration

Git definitions describe the behavior: agents, skills, triggers, task templates, and model catalog entries.

Execution

Runners pull tasks, create workspaces, start Docker/gVisor containers, and drive OpenCode, Claude Code, or Pi.

Evidence

Events, exports, artifacts, and audit rows make agent behavior inspectable after the run.

Mental model: Chetter separates policy, queueing, and evidence from the harness that actually talks to the LLM.

Domain model

The vocabulary

These are the core nouns you see in the database, MCP tools, web UI, and logs.

Task

A queued unit of work: prompt, repo/ref, image, harness, model, status, attempts, lease, result.

Session

Long-lived identity and workspace lineage for an agent. Can be completed, paused, recoverable, or expired.

Session Run

One prompt execution inside a session. Every submitted task creates one run.

Runner

A worker process with slots. It registers health, claims tasks, renews leases, and owns checkpoints.

Trigger

Automation rule that creates tasks: cron, PR review, issue/comment, or manually invoked schedule.

Definition

Git-backed behavior file: agent, skill, trigger, or task template, stored with commit and content hash.

Event

Task timeline record: progress, heartbeat, completion, or classified failure payload.

Artifact

GitHub issue, PR, comment, review, or definition proposal created by a task and linked back to it.

Git is authoritative

Agent behavior changes through normal pull requests, review, diffs, rollback, blame, and branch experimentation.

TiDB is the runtime index

Chetter syncs, validates, hashes, and materializes active definitions for fast lookup and historical traceability.

Images stay stable

Toolchains and harnesses live in runner images. Prompts, skills, agents, triggers, and model choices live in Git.

Agents can improve config

Meta-agents inspect prior exports, propose definition changes, and open PRs instead of mutating production behavior silently.

Definitions repo

A trigger is just reviewed YAML

triggers/example-daily-review.yaml
name: example-daily-review

What this becomes

  • A row in definitions with type trigger, source commit, path, and content hash.
  • A runtime trigger in chetter_triggers, owned by the default definition source.
  • A future task template: prompt, repo, agent, skills, timeout, model, harness, and session settings.

Important: trigger YAML can reference agent and skill names from the same config repo, keeping automation behavior reviewable as a unit.

Trigger mechanisms

How tasks enter the queue

Cron

Runs on a cron expression or descriptor like @hourly. Good for nightly docs, scans, maintenance, reports.

PR review

GitHub webhook matches a repository and asks an agent to review a PR or respond to /chetter-review.

Issue/comment

Issue events and comments can create implementation, triage, or response tasks, optionally gated by labels.

Manual MCP

chetter_submit_task creates one-off tasks from any MCP client, including OpenCode and Claude Code.

Session resume

chetter_resume_agent_session creates a follow-up task pinned to the runner that owns the workspace/checkpoint.

Event callback

Task events can trigger follow-up tasks, webhooks, or Slack notifications for simple workflow chaining.

Why the pull model matters

  • Scale runners horizontally without assigning work by hand.
  • Use leases so crashed runners do not strand tasks forever.
  • Support runner affinity for resumable sessions via required_runner_id.
  • Drain a runner during rollout without killing in-flight tasks.

Walkthrough

The life of a task/session

1Create

MCP, trigger, webhook, or resume creates a task.

2Materialize

Task, agent session, and first session run are stored.

3Claim

A runner atomically claims with SKIP LOCKED.

4Execute

Runner starts sandbox and harness in a workspace.

5Observe

Events, heartbeat, logs, and transcript stream back.

6Finish

Task completes, fails, pauses, or becomes recoverable.

Step 1

Creation creates three linked records

Task
task_*
queue item and status
Agent Session
sess_*
lineage and resume identity
Session Run
run_*
one prompt execution

Stored on submit

  • Prompt, repo URL/ref, timeout, skills, non-secret env.
  • Agent image, harness, provider/model/variant overrides.
  • Trigger name/type and team id when created by automation.
  • Session mode, pause reason, TTL, and runner affinity fields.

Default one-shot tasks still get session records, so history and UI lineage are consistent.

Step 2

Claiming is atomic and lease-backed

Claim

Runner asks for work. The server locks the oldest eligible pending task with FOR UPDATE SKIP LOCKED.

Lease

Claim sets runner_id, claimed_at, lease_expires_at, and increments attempts.

Affinity

Resumed work can require a specific runner, so only the owner of the workspace/checkpoint can claim it.

Design effect: many runners can poll at once, but only one gets a given task. If that runner disappears, the lease expires and the reaper can recover the task.

Step 3

The runner builds the workspace bubble

Before the harness runs

  1. Clone git_url at git_ref.
  2. Create or reuse a workspace under the runner workspace root.
  3. Resolve model/provider details from the active model catalog.
  4. Generate harness config and MCP bridge config.
  5. Start Docker/gVisor task container when using isolated mode.

Injected into the task

CHETTER_TASK_ID WORKSPACE MCP_SOCKET_PATH GITHUB_TOKEN CHETTER_MODEL_ID

The agent can run commands inside its sandbox and gets controlled MCP tools for workspace access and audited GitHub writes.

Step 4

The harness does the agent work

OpenCode

Richest integration. Runs in HTTP serve mode, supports streaming events, session export, and task-container isolation.

Claude Code

Batch subprocess mode. Useful when a simpler one-shot CLI execution path is enough.

Pi

JSONL RPC subprocess. Provider-rich harness with streaming control and session messages.

Key boundary: Chetter coordinates execution and records evidence. The harness decides how to interact with the LLM and tools inside the workspace.

Step 5

Events make the run observable

Two channels of liveness

Heartbeat
Renews the task lease. Stored at most once per minute to avoid event spam.

Progress
Harness events, tool calls, summaries, errors, and completion payloads.

Both feed the web UI, MCP status tools, distilled progress timelines, and post-run debugging.

task timeline
task.progress      agent started
Step 6

Completion records results and artifacts

Task result

  • done, error, or cancelled status.
  • Summary, classified error category, timing, tokens, and cost fields.
  • Markdown session export for later inspection or recovery context.

GitHub artifacts

  • Agents use Chetter MCP tools for PRs, issues, comments, and reviews.
  • Chetter adds a canonical footer and records chetter_task_artifacts.
  • Artifacts link task, session, run, repository, number, URL, branch, and SHA.

Why it matters: weeks later, you can answer which task created a PR, which trigger created the task, which agent config was active, and what the agent saw.

One-shot

Session completes when the first run completes.

Paused

Successful resumable run preserves workspace and waits for follow-up.

Recoverable

Timeout/transport failures can keep enough state to resume manually.

Reliability

The reaper closes the loop

What it watches

  • Expired leases when runner heartbeats stop.
  • Tasks that exceed timeout_sec.
  • Tasks silent long enough to be considered stale.
  • Paused sessions whose TTL has expired.

What it does

  • Retries eligible tasks up to max_attempts.
  • Marks exhausted tasks terminal with a clear error category.
  • Expires stale paused sessions and checkpoints.
  • Keeps the queue moving without operator intervention.

Failure is data. Chetter records the error, category, run, runner, and transcript/export when available so recovery and improvement are possible.

Putting it together

The pedagogical shortcut

Remember this

Config in Git decides what should happen.

TiDB remembers what is happening and what happened.

Runners do the risky work in isolated workspaces.

Artifacts make outcomes traceable in GitHub.

Task/session lifecycle

  1. Create task, session, and run.
  2. Runner claims task with a renewable lease.
  3. Runner creates workspace and starts harness.
  4. Events and heartbeats stream back.
  5. Results, exports, artifacts, and audit rows are stored.
  6. Session completes, pauses, or becomes recoverable.
